SECTION 1 – MICROSOFT DEFENDER FOR ENDPOINT (MDE)

1.1 Onboarding Verification

# Check MDE sensor status
sc query sense
Get-Service -Name "sense"

# Check connectivity (run on endpoint)
# Download: https://aka.ms/mdeclientanalyzer
MDATPClientAnalyzer.cmd

# Manual onboarding check
reg query "HKLM\SOFTWARE\Microsoft\Windows Advanced Threat Protection\Status" /v OnboardingState
# Value 1 = onboarded

# Check MDE in event log
wevtutil qe "Microsoft-Windows-SENSE/Operational" /c:10 /f:text

# Force detection test (EICAR-like)
powershell.exe -NoExit -ExecutionPolicy Bypass -WindowStyle Hidden (New-Object System.Net.WebClient).DownloadFile('https://secure.eicar.org/eicar.com','C:\temp\eicar.com')

1.2 MDE Troubleshooting

  • Device not appearing in portal: Check onboarding script ran, sense service running, network connectivity to *.securitycenter.windows.com
  • Alerts not generating: Check policy assignments, exclusions too broad, sensor health
  • Performance impact: Check exclusions for known applications (SQL, Exchange, backup agents)
  • EDR in block mode: Available even if 3rd party AV is primary — blocks missed threats

1.3 Attack Surface Reduction (ASR) Rules

  • Block executable content from email client and webmail
  • Block Office applications from creating child processes
  • Block Office applications from injecting code into other processes
  • Block JavaScript or VBScript from launching downloaded executable content
  • Block execution of potentially obfuscated scripts
  • Block credential stealing from Windows LSASS
  • Block process creations from PSExec and WMI commands
# Check ASR rules via PowerShell
Get-MpPreference | Select AttackSurfaceReductionRules_Ids, AttackSurfaceReductionRules_Actions

# ASR rule actions: 0=Disabled, 1=Block, 2=Audit, 6=Warn

SECTION 2 – MICROSOFT DEFENDER FOR IDENTITY

2.1 Defender for Identity Monitoring

  • Detects: Pass-the-Hash, Pass-the-Ticket, Golden Ticket, DCSync, brute force, lateral movement
  • Sensor installed on DCs — reads AD traffic and Windows events
  • Requires: .NET 4.7+, port 443 to *.atp.azure.com, event IDs 4776, 4732, 4733, 4728, 4729, 8004

SECTION 3 – MICROSOFT SENTINEL (SIEM)

3.1 Essential KQL Queries

// Failed sign-ins from external IPs
SigninLogs
| where ResultType != "0"
| where IPAddress !startswith "10." and IPAddress !startswith "192.168."
| summarize FailureCount=count() by UserPrincipalName, IPAddress, ResultDescription
| where FailureCount > 5
| order by FailureCount desc

// Impossible travel detection
SigninLogs
| where ResultType == "0"
| summarize Locations=make_set(Location), LocationCount=dcount(Location) by UserPrincipalName, bin(TimeGenerated, 1h)
| where LocationCount > 1

// Suspicious PowerShell execution
SecurityEvent
| where EventID == 4688
| where CommandLine has_any ("Invoke-Mimikatz","Invoke-Expression","DownloadString","EncodedCommand","-enc")
| project TimeGenerated, Computer, Account, CommandLine

// MFA fatigue / push spam
SigninLogs
| where ResultType == "50074" // MFA required
| summarize Attempts=count() by UserPrincipalName, bin(TimeGenerated, 10m)
| where Attempts > 5

// Mailbox forwarding rules (data exfil indicator)
OfficeActivity
| where Operation == "New-InboxRule" or Operation == "Set-InboxRule"
| where Parameters has "ForwardTo" or Parameters has "RedirectTo"
| project TimeGenerated, UserId, Operation, Parameters

3.2 Sentinel Best Practices

  • Connect all M365 data connectors (Azure AD, O365, MDE, Defender for Cloud)
  • Enable UEBA (User and Entity Behavior Analytics)
  • Tune analytics rules — reduce false positives before going to production
  • Use Watchlists for VIP users, critical assets, known IOCs
  • Automate with Playbooks (Logic Apps) for common response actions

SECTION 4 – ZERO TRUST ARCHITECTURE

4.1 Zero Trust Principles

  1. Verify explicitly: Always authenticate and authorize based on all available data points
  2. Use least privilege access: Limit user access with JIT/JEA, risk-based adaptive policies
  3. Assume breach: Minimize blast radius, segment access, verify E2E encryption, use analytics

4.2 Zero Trust Implementation Checklist (Microsoft Stack)

SECTION 5 – MICROSOFT PURVIEW (COMPLIANCE)

5.1 Data Loss Prevention (DLP)

  • Protect sensitive data (SSN, credit cards, health records) across Exchange, SharePoint, Teams, endpoints
  • Policy tip: Warn users before they share sensitive content
  • Block: Prevent sharing with external users
  • Override: Allow with business justification

5.2 Sensitivity Labels

  • Public, Internal, Confidential, Highly Confidential
  • Apply encryption and access restrictions
  • Auto-labeling based on content inspection
  • Container labels for Teams, SharePoint sites, M365 groups

5.3 Retention Policies

  • Retain and/or delete content after specified period
  • Apply to Exchange, SharePoint, OneDrive, Teams, Yammer
  • Retention labels for item-level control
  • Disposition review for regulated content

SECTION 6 – INCIDENT RESPONSE PLAYBOOK

6.1 Compromised Account Response

  1. Reset password immediately
  2. Revoke all refresh tokens: Revoke-MgUserSignInSession -UserId user@domain.com
  3. Check and remove mailbox forwarding rules
  4. Check and remove OAuth app consents
  5. Review sign-in logs for scope of compromise
  6. Check Inbox rules, delegates, mobile device access
  7. Enable MFA if not already enabled
  8. Notify user and document incident

6.2 Ransomware Response

  1. Isolate affected machines (disconnect from network, NOT power off)
  2. Identify ransomware variant (ransom note, encrypted file extension)
  3. Check scope: which machines, which shares, which backup data affected
  4. Preserve evidence: memory dumps, event logs, network logs
  5. Check backup integrity BEFORE restoring
  6. Identify entry point: phishing email, RDP brute force, vulnerability exploit
  7. Remediate entry point before restoring
  8. Restore from clean backup
  9. Rotate ALL credentials (domain admin, service accounts, krbtgt)
  10. Post-incident review and documentation

6.3 Business Email Compromise (BEC) Response

  1. Secure the account (reset password, revoke sessions, enable MFA)
  2. Search message trace for phishing emails sent FROM the account
  3. Check Inbox rules for hidden forwarding
  4. Check OAuth consents for malicious apps
  5. Notify recipients of fraudulent emails
  6. If financial fraud attempted: notify bank immediately, file police report
  7. Review Unified Audit Log for all account activity during compromise

SECTION 7 – SECURE SCORE PRIORITIES

  • Enable MFA for all admins (Critical)
  • Block legacy authentication (Critical)
  • Enable self-service password reset (High)
  • Don't expire passwords (use banned password list instead) (High)
  • Protect all users with sign-in risk policy (High)
  • Require MFA for all users (High)
  • Enable Microsoft Defender for Office 365 Safe Attachments (High)
  • Enable audit logging (Critical)
  • Create DLP policies (Medium)
  • Enable privileged identity management (High)
