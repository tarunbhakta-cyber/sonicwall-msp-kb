SonicWall MSP Knowledge Base – Master Reference

Purpose: This document defines authoritative security, deployment, and operational standards for managed SonicWall TZ and NSa firewall environments.

SECTION 1 – MSP GLOBAL SECURITY BASELINE (MANDATORY)

1.1 Administrative Security

  • WAN management disabled unless explicitly required and IP restricted.
  • Default admin account renamed or disabled.
  • Strong password policy enforced.
  • MFA required for all administrative access.
  • Management access restricted to trusted IP ranges.

1.2 SSL VPN Mandatory Controls

  • MFA required for all SSL VPN users.
  • TLS 1.2+ only. Legacy ciphers disabled.
  • GeoIP restrictions enforced.
  • Login attempt lockout configured.
  • Botnet filter enabled.
  • Brute-force detection alerts enabled.

1.3 Firmware Policy

  • Quarterly firmware review.
  • Critical security patches deployed within 7 days.
  • Firmware tested in lab before production deployment.
  • Config backup taken before and after upgrade.

1.4 Logging & Monitoring

  • Logs exported to centralized SIEM.
  • Critical events trigger alerting.
  • VPN login monitoring enabled.
  • Configuration change alerts enabled.

SECTION 2 – TZ SERIES DEPLOYMENT STANDARDS

2.1 Performance Guardrails

  • Sustained CPU above 65% requires investigation.
  • DPI-SSL enabled selectively based on performance impact.
  • Heavy App Control policies used cautiously.
  • Concurrent VPN user limits monitored.

2.2 Recommended Security Profile

  • IPS enabled.
  • Gateway AV enabled.
  • Anti-Spyware enabled.
  • Botnet filter enabled.
  • GeoIP filtering enforced.

2.3 Common TZ Risk Patterns

  • Any-Any firewall rules.
  • DPI-SSL enabled without exclusions.
  • WAN management exposed.
  • Weak site-to-site VPN pre-shared keys.

SECTION 3 – NSa SERIES DEPLOYMENT STANDARDS

3.1 Advanced Security Posture

  • Full DPI-SSL recommended unless performance impact exceeds 20%.
  • Segmentation strongly encouraged.
  • Client certificate enforcement for VPN preferred.
  • HA Active/Standby deployment recommended.

3.2 High Throughput Design

  • Monitor DPI memory utilization.
  • Review connection count thresholds.
  • Enable full security suite where hardware allows.

SECTION 4 – MSP OPERATIONAL PLAYBOOKS

4.1 New Client Onboarding

  1. Baseline configuration applied.
  2. Security services enabled.
  3. SSL VPN hardened.
  4. Logs integrated with SIEM.
  5. Firmware updated to approved version.

4.2 Incident Response – Suspected Compromise

  1. Disable affected VPN accounts.
  2. Review admin login logs.
  3. Export configuration for review.
  4. Validate firmware integrity.
  5. Rotate VPN shared secrets.

4.3 Annual Review Checklist

  • HA failover test completed.
  • Firewall rule audit performed.
  • VPN account audit performed.
  • GeoIP restrictions reviewed.
  • Security services license validated.

SECTION 6 – WINDOWS NETWORKING ARCHITECTURE STANDARDS

6.1 Active Directory & DNS Design Principles

  • Domain controllers must use internal DNS only.
  • Firewall must allow required AD ports between sites.
  • Time synchronization must use domain hierarchy.
  • Split DNS properly designed for VPN clients.
  • Secure dynamic updates enforced.

6.2 Required AD Ports (Must Be Allowed Through Firewall)

  • TCP/UDP 53 – DNS
  • TCP/UDP 88 – Kerberos
  • TCP 135 – RPC
  • TCP 389 – LDAP
  • TCP 445 – SMB
  • TCP 3268 – Global Catalog
  • Ephemeral RPC range 49152–65535

6.3 DNS Troubleshooting Standards

  • Never point domain-joined systems to ISP DNS.
  • Verify forward and reverse lookup zones.
  • Check AD-integrated zone replication.
  • Use nslookup and dcdiag for validation.

6.4 Site-to-Site VPN + AD Replication

  • VPN must allow full RPC range.
  • MTU issues must be tested.
  • Fragmented Kerberos packets monitored.
  • Latency impact on replication reviewed.

6.5 Windows Server Core Networking Knowledge

  • Routing table analysis.
  • ARP table validation.
  • NIC binding order review.
  • Windows Firewall advanced rules.
  • Netsh and PowerShell diagnostics.

6.6 Common MSP Windows Networking Failure Patterns

  • Split tunnel misconfiguration.
  • Improper DNS forwarders.
  • Firewall blocking dynamic RPC.
  • MTU mismatch across VPN.
  • Asymmetric routing via SonicWall.

6.7 Troubleshooting Methodology (Authoritative)

  1. Validate physical connectivity.
  2. Validate IP configuration.
  3. Validate DNS resolution.
  4. Validate routing table.
  5. Validate firewall logs.
  6. Validate AD replication health.
