SECTION 1 – MSP GLOBAL SECURITY BASELINE (MANDATORY)

1.1 Administrative Security

  • WAN management disabled unless explicitly required and IP restricted.
  • Default admin account renamed or disabled.
  • Strong password policy enforced.
  • MFA required for all administrative access.
  • Management access restricted to trusted IP ranges.

1.2 SSL VPN Mandatory Controls

  • MFA required for all SSL VPN users.
  • TLS 1.2+ only. Legacy ciphers disabled.
  • GeoIP restrictions enforced.
  • Login attempt lockout configured.
  • Botnet filter enabled.
  • Brute-force detection alerts enabled.

1.3 Firmware Policy

  • Quarterly firmware review.
  • Critical security patches deployed within 7 days.
  • Firmware tested in lab before production deployment.
  • Config backup taken before and after upgrade.

1.4 Logging & Monitoring

  • Logs exported to centralized SIEM.
  • Critical events trigger alerting.
  • VPN login monitoring enabled.
  • Configuration change alerts enabled.

SECTION 6 – WINDOWS NETWORKING ARCHITECTURE STANDARDS

6.1 Active Directory & DNS Design Principles

  • Domain controllers must use internal DNS only.
  • Firewall must allow required AD ports between sites.
  • Time synchronization must use domain hierarchy.
  • Split DNS properly designed for VPN clients.
  • Secure dynamic updates enforced.

6.2 Required AD Ports (Must Be Allowed Through Firewall)

  • TCP/UDP 53 – DNS
  • TCP/UDP 88 – Kerberos
  • TCP 135 – RPC
  • TCP 389 – LDAP
  • TCP 445 – SMB
  • TCP 3268 – Global Catalog
  • Ephemeral RPC range 49152–65535

6.3 DNS Troubleshooting Standards

  • Never point domain-joined systems to ISP DNS.
  • Verify forward and reverse lookup zones.
  • Check AD-integrated zone replication.
  • Use nslookup and dcdiag for validation.

6.4 Site-to-Site VPN + AD Replication

  • VPN must allow full RPC range.
  • MTU issues must be tested.
  • Fragmented Kerberos packets monitored.
  • Latency impact on replication reviewed.

6.5 Windows Server Core Networking Knowledge

SECTION 7 – MICROSOFT 365 / EXCHANGE ONLINE

7.1 Tenant Architecture & Licensing

  • One tenant per organization. Multi-tenant management via Microsoft 365 Lighthouse or partner center for MSPs.
  • Use Microsoft 365 Business Premium as baseline for SMB clients (includes Intune, Defender for Business, Entra ID P1).
  • Enterprise clients: E3 for core productivity + compliance; E5 for advanced security, compliance, and analytics.
  • F1/F3 licenses for frontline workers (kiosks, shared devices, deskless staff).
  • Exchange Online Plan 1 included in Business Basic/Standard/Premium; Plan 2 adds archiving, legal hold, unlimited storage.
  • Track license assignment with Get-MgUser -All | Select DisplayName, AssignedLicenses — never let unused licenses accumulate.

7.2 Mail Flow Architecture

  • MX records must point to Exchange Online Protection (EOP): <tenant>.mail.protection.outlook.com.
  • If using a third-party gateway (Mimecast, Proofpoint, Barracuda), configure Enhanced Filtering for Connectors (skip listing) so EOP sees the true source IP.
  • Outbound mail routes through EOP by default. Custom connectors only needed for smart host routing, partner orgs, or on-prem relay.
  • Hybrid mail flow: Centralized (routes through on-prem) vs. Direct (each routes independently). Prefer direct for new deployments.
  • Internal relay: Use receive connectors with TLS + certificate-based auth. Never allow anonymous relay to the internet.

7.2.1 Transport Rules (Mail Flow Rules)

  • Use transport rules for disclaimers, external sender warnings, attachment blocking, and encryption triggers.
  • External email warning banner: Apply to all inbound mail from outside the organization — prepend HTML banner.
  • Block auto-forwarding to external recipients: Transport rule with condition "message type = auto-forward" → reject.
  • Encryption: Trigger Microsoft Purview Message Encryption via transport rule when subject contains "[Encrypt]" or sensitivity labels applied.
  • Rule processing order matters — lower priority number = higher precedence. Document all rules in a runbook.

7.3 Email Authentication — SPF, DKIM, DMARC

Critical: All three records (SPF + DKIM + DMARC) must be configured for every domain. Missing any one creates spoofing and deliverability risks.

7.3.1 SPF (Sender Policy Framework)

  • DNS TXT record on the domain root. Authorizes which servers may send email for the domain.
  • Standard M365 SPF: v=spf1 include:spf.protection.outlook.com -all
  • If using third-party services (Mailchimp, SendGrid, etc.), add their include: directives.
  • Hard fail (-all) preferred over soft fail (~all) once all legitimate senders are identified.
  • SPF has a 10-DNS-lookup limit. Use include: sparingly; flatten if approaching the limit.
  • Validate with: nslookup -q=txt domain.com or Resolve-DnsName domain.com -Type TXT

7.3.2 DKIM (DomainKeys Identified Mail)

  • Cryptographically signs outbound email. Prevents tampering in transit.
  • Enable in Microsoft 365 Defender portal → Email & collaboration → Policies → DKIM.
  • Two CNAME records required per domain:
    • selector1._domainkey.domain.com → selector1-domain-com._domainkey.tenantname.onmicrosoft.com
    • selector2._domainkey.domain.com → selector2-domain-com._domainkey.tenantname.onmicrosoft.com
  • After DNS propagation, enable DKIM signing in the portal. Rotation is automatic (M365 rotates keys).
  • Third-party senders (Mailchimp, etc.) need their own DKIM configuration — each service provides separate CNAME/TXT records.

7.3.3 DMARC (Domain-based Message Authentication, Reporting & Conformance)

  • DNS TXT record: _dmarc.domain.com
  • Start with monitoring: v=DMARC1; p=none; rua=mailto:dmarc@domain.com; ruf=mailto:dmarc@domain.com
  • After 2-4 weeks of monitoring, move to quarantine: p=quarantine; pct=25 → gradually increase pct to 100.
  • Final enforcement: p=reject — unauthorized mail is rejected outright.
  • Use a DMARC reporting service (Valimail, dmarcian, EasyDMARC) to analyze aggregate reports.
  • Subdomain policy: sp=reject to prevent subdomain spoofing.

7.4 Shared Mailboxes & Resource Mailboxes

  • Shared mailboxes: No license required if under 50 GB. Auto-licensed at 50 GB+.
  • Do not assign passwords to shared mailboxes — block sign-in. Access only via delegation.
  • Send-As vs. Send-on-Behalf: Send-As hides the sender's identity; Send-on-Behalf shows "sent by X on behalf of Y".
  • Resource mailboxes (Room/Equipment): Configure booking policies — auto-accept, conflict resolution, booking delegates.
  • PowerShell: New-Mailbox -Shared -Name "Support" -PrimarySmtpAddress support@domain.com
  • Convert user mailbox to shared: Set-Mailbox -Identity user@domain.com -Type Shared — then remove license.

7.5 Retention Policies & Litigation Hold

  • Microsoft Purview retention policies: Apply to Exchange, SharePoint, OneDrive, Teams, Yammer.
  • Retention = keep + optional delete. Labels can be auto-applied based on sensitive info types or trainable classifiers.
  • Litigation Hold: Preserves all mailbox content (including deleted items) indefinitely or for a specified duration. Requires Exchange Online Plan 2 or E3+.
  • In-Place Hold (legacy) → replaced by Microsoft Purview retention. Migrate existing holds.
  • eDiscovery: Content Search for ad-hoc searches; eDiscovery (Standard) for case management; eDiscovery (Premium) for advanced analytics, review sets, custodian management.
  • Default MRM policy: Deletes items from Recoverable Items after 14 days. Extend to 30 days for compliance: Set-Mailbox -SingleItemRecoveryEnabled $true -RetainDeletedItemsFor 30

7.6 Data Loss Prevention (DLP)

  • DLP policies detect and protect sensitive information (SSN, credit cards, health records) across Exchange, SharePoint, OneDrive, Teams, and endpoints.
  • Use built-in sensitive information types (SITs) — 300+ available. Create custom SITs for org-specific data (account numbers, project codes).
  • Policy tips: Warn users before they share sensitive data. Configure in policy settings.
  • Incident reports: Auto-email to compliance team when DLP rule is triggered.
  • Start with "test mode" (audit only) → review matches → enable enforcement.
  • Endpoint DLP: Extends DLP to Windows/macOS devices. Requires Intune enrollment or onboarding to Defender for Endpoint.

7.7 Anti-Spam & Anti-Phishing Policies (EOP)

  • Default policies provide baseline protection. Create custom policies for stricter controls per group.
  • Anti-phishing: Enable mailbox intelligence, spoof intelligence, impersonation protection (protect priority accounts by name + domain).
  • Safe Attachments (Defender for O365): Detonates attachments in a sandbox. Enable "Dynamic Delivery" for minimal user disruption.
  • Safe Links (Defender for O365): URL rewriting + time-of-click verification. Enable for email, Teams, and Office apps.
  • Quarantine policies: Define who can release quarantined messages. Admin-only for high-confidence phish; user self-service for spam.
  • Tenant Allow/Block List: Override false positives/negatives. Block specific senders, URLs, or file hashes.
  • Priority account protection: Tag executive accounts for enhanced detection and dedicated alerting.

7.8 Hybrid Exchange Configuration

  • Hybrid Configuration Wizard (HCW): Run from on-prem Exchange to establish hybrid. Minimum on-prem: Exchange 2016 CU23+ or 2019 CU12+.
  • Azure AD Connect required for directory synchronization (users, groups, contacts).
  • Free/Busy sharing: Requires Organization Relationship + OAuth authentication between on-prem and EXO.
  • Mailbox migrations: Use Migration Batches (New-MigrationBatch). Cutover for <150 mailboxes; Staged or Hybrid for larger orgs.
  • Mail flow: Centralized transport (all outbound through on-prem) vs. Direct delivery. Direct is simpler and preferred for most MSP clients.
  • Decommissioning on-prem: Keep at least one Exchange server for recipient management until fully migrated to cloud-only management (requires Exchange Management Tools on a workstation).

7.9 Troubleshooting Exchange Online

  1. Message trace: Microsoft 365 Defender → Email & collaboration → Exchange message trace. Search by sender, recipient, date range, status.
  2. NDR analysis: Check enhanced status codes (5.7.1 = relay denied, 5.1.1 = invalid recipient, 4.4.1 = timeout).
  3. Mail flow issues: Verify MX, SPF, connectors, transport rules. Use Remote Connectivity Analyzer (testconnectivity.microsoft.com).
  4. Outlook connectivity: Run Microsoft Support and Recovery Assistant (SaRA). Check Autodiscover: Test-OutlookConnectivity.
  5. Delegation not working: Verify permissions with Get-MailboxPermission and Get-RecipientPermission. Automapping issues → remove and re-add with -AutoMapping $false.
  6. Calendar issues: Get-CalendarProcessing for room mailbox settings; Test-CalendarConnectivity for hybrid free/busy.

SECTION 8 – AZURE AD / MICROSOFT ENTRA ID

8.1 Entra ID Architecture Overview

  • Microsoft Entra ID (formerly Azure AD) is the cloud identity provider for all Microsoft 365, Azure, and SaaS integrations.
  • Entra ID Free: Included with M365. SSO, basic MFA, self-service password reset (cloud-only).
  • Entra ID P1 (included in M365 Business Premium, E3): Conditional Access, group-based licensing, dynamic groups, on-prem writeback.
  • Entra ID P2 (included in E5): Privileged Identity Management (PIM), Identity Protection, access reviews.
  • Each tenant has a .onmicrosoft.com domain + custom verified domains. Verify domains via TXT record in DNS.
  • Object limits: Free = 500k objects. P1/P2 = unlimited. Monitor with Get-MgOrganization | Select DirectorySizeQuota.

8.2 Conditional Access Policies

Conditional Access is the primary security control plane for Microsoft cloud. Every MSP client should have a baseline set of CA policies deployed from day one.

8.2.1 Mandatory Baseline Policies

  1. Require MFA for all users — Target: All users. Exclude: Break-glass accounts. Grant: Require MFA.
  2. Require MFA for admin roles — Target: Directory roles (Global Admin, Exchange Admin, etc.). Grant: Require MFA + compliant device.
  3. Block legacy authentication — Target: All users. Condition: Client apps = Exchange ActiveSync, Other clients. Grant: Block.
  4. Require compliant device for Office apps — Target: All users. Cloud apps: Office 365. Grant: Require device compliance (Intune).
  5. Block sign-in from untrusted countries — Target: All users. Condition: Named locations = all except allowed countries. Grant: Block.
  6. Require app protection for mobile — Target: All users. Condition: Device platform = iOS, Android. Grant: Require approved client app or app protection policy.

8.2.2 Conditional Access Design Principles

  • Always maintain 2 break-glass accounts excluded from all CA policies. Cloud-only, no MFA, strong 20+ character passwords stored in a physical safe.
  • Use "Report-only" mode for 7-14 days before enforcing a new policy. Review sign-in logs for unintended blocks.
  • Name policies consistently: CA001 - Grant - Require MFA for All Users, CA002 - Block - Legacy Authentication.
  • Use named locations for trusted IPs (office networks, VPN exit points).
  • Session controls: Sign-in frequency (re-auth every X hours), persistent browser session, Conditional Access App Control (MCAS proxy).
  • Device filters: Target specific hardware, OS versions, or compliance states.

8.3 Multi-Factor Authentication (MFA)

  • Entra ID MFA methods (strongest to weakest): FIDO2 security key → Windows Hello for Business → Microsoft Authenticator (passwordless) → Authenticator push notification → OATH hardware token → SMS/Voice call.
  • Security defaults: Free-tier MFA for all users. Disable once Conditional Access is configured (they conflict).
  • Number matching: Enabled by default for Authenticator push. Prevents MFA fatigue attacks.
  • Per-user MFA (legacy portal): Do not use. Migrate to Conditional Access-based MFA.
  • Registration campaigns: Force users to set up Authenticator via Authentication methods → Registration campaign.
  • Temporary Access Pass (TAP): One-time or time-limited passcode for onboarding users to passwordless methods. Configure in Authentication Methods.

8.4 Hybrid Identity — Azure AD Connect / Cloud Sync

  • Azure AD Connect (legacy): Installed on-prem, syncs AD DS objects to Entra ID. Supports Password Hash Sync (PHS), Pass-Through Authentication (PTA), and Federation (ADFS).
  • Entra Cloud Sync (modern): Lightweight agent-based sync. Supports multi-forest, no need for dedicated server. Preferred for new deployments.
  • Password Hash Sync (PHS): Recommended. Hashes synced to cloud. Enables leaked credential detection, seamless SSO.
  • Pass-Through Authentication (PTA): Validates passwords against on-prem AD in real-time. Use when regulatory requirements prohibit cloud password storage.
  • Seamless SSO: Auto-signs in domain-joined users on the corporate network. Requires computer account in AD.
  • Password writeback: Allows cloud SSPR to write password changes back to on-prem AD. Requires P1+ license.
  • Attribute filtering: Sync only required OUs. Never sync service accounts, admin accounts, or test objects.
  • Monitoring: Azure AD Connect Health (P1 required) provides sync error alerts, performance monitoring.

8.5 Application Registrations & Enterprise Apps

  • App registrations: Developer-facing configuration (client ID, redirect URIs, certificates/secrets, API permissions).
  • Enterprise apps: Admin-facing view (user assignments, SSO config, provisioning, Conditional Access targeting).
  • SSO protocols: SAML 2.0, OIDC/OAuth 2.0, Header-based, Password-based. Prefer OIDC for new integrations.
  • User consent: Block by default. Require admin consent for all third-party apps. Configure in Enterprise apps → Consent and permissions.
  • SCIM provisioning: Automate user lifecycle (create, update, disable) to SaaS apps. Supported for ServiceNow, Salesforce, Slack, etc.
  • Certificate rotation: Track expiry of SAML signing certificates and app secrets. Set calendar reminders 30 days before expiry.
  • Managed identities: For Azure resources (VMs, App Services) — eliminates credential management. Use system-assigned for single-resource, user-assigned for shared identity.

8.6 Privileged Identity Management (PIM)

  • Requires Entra ID P2 license.
  • Just-in-time activation: Admins request role elevation for limited duration (1-24 hours).
  • Require approval for critical roles: Global Admin, Exchange Admin, Security Admin.
  • Require MFA + justification for all role activations.
  • Access reviews: Quarterly review of permanent role assignments. Auto-remove if not reconfirmed.
  • Eligible vs. Active assignments: Use eligible for all roles. Active (permanent) only for break-glass accounts.
  • Alert on suspicious activations: PIM sends alerts for unusual patterns.

8.7 Identity Protection

  • Requires Entra ID P2.
  • User risk: Detects compromised credentials (leaked password databases, impossible travel, malware-linked sign-ins).
  • Sign-in risk: Real-time evaluation of each authentication attempt (unfamiliar location, anonymous IP, atypical travel).
  • Risk-based Conditional Access: Low risk → allow; Medium → require MFA; High → block + force password reset.
  • Risky users report: Review weekly. Confirm compromised, dismiss false positives, or force password reset.
  • Risk detections feed into Microsoft Sentinel for SIEM correlation.

8.8 Entra ID Troubleshooting

  1. Sign-in logs: Entra ID → Monitoring → Sign-in logs. Filter by user, app, status, CA policy, risk level.
  2. Audit logs: Track admin actions (password resets, role assignments, policy changes).
  3. CA policy evaluation: Sign-in log → Conditional Access tab shows which policies applied/failed/not-applied and why.
  4. "What If" tool: Simulate CA policy evaluation for a specific user + app + conditions before deploying.
  5. Token issues: Check token lifetime policies. Default access token = 1 hour, refresh token = 90 days (sliding).
  6. Guest access issues: Check B2B collaboration settings, cross-tenant access policies, and CA policies targeting "Guest or external users".

SECTION 9 – WINDOWS SERVER & ACTIVE DIRECTORY

9.1 Active Directory Domain Services (AD DS) Design

  • Single-domain, single-forest is standard for SMB MSP clients. Multi-domain only for regulatory isolation or mergers.
  • Functional levels: Raise to Windows Server 2016+ to enable Privileged Access Management, time-based group membership.
  • Domain naming: Use a subdomain of a public domain (e.g., ad.company.com) — never .local or single-label domains.
  • Minimum 2 DCs per domain for redundancy. Place a DC at each major site with 50+ users.
  • AD Recycle Bin: Enable immediately on all domains. Enable-ADOptionalFeature 'Recycle Bin Feature' -Scope ForestOrConfigurationSet -Target 'forest.com'
  • Fine-Grained Password Policies (PSOs): Use for privileged accounts. Minimum 16 characters, 90-day max age, no password reuse for 24 generations.

9.3 Group Policy (GPO) Standards

  • Naming convention: [OU] - [Category] - [Description] (e.g., Workstations - Security - BitLocker Enforcement).
  • Never modify the Default Domain Policy except for: Password Policy, Account Lockout Policy, Kerberos Policy.
  • Never modify the Default Domain Controllers Policy except for: Audit Policy, User Rights Assignment for DCs.
  • Delegation: Use separate GPOs per function. One GPO per setting area is easier to troubleshoot than monolithic GPOs.
  • WMI filters: Use sparingly — they execute on every GPO processing cycle and slow logon/startup.
  • Security filtering: Prefer security group filtering over WMI where possible.
  • Loopback processing: Use "Replace" mode for kiosk/terminal servers. "Merge" for conference room PCs.
  • Central Store: Create in \\domain\SYSVOL\domain\Policies\PolicyDefinitions for consistent ADMX template management.

9.4 DNS Server Standards

  • AD-integrated zones: Always use AD-integrated for domain zones. Provides secure dynamic updates and multi-master replication.
  • Forwarders: Point to reliable external resolvers (Cloudflare 1.1.1.1, Google 8.8.8.8, Quad9 9.9.9.9). Never use ISP DNS as primary.
  • Conditional forwarders: Use for partner domains, Azure Private DNS zones, or split-brain DNS scenarios.
  • Scavenging: Enable aging and scavenging. No-refresh interval: 7 days. Refresh interval: 7 days. Scavenging period: 7 days. Enable on at least one DC.
  • Reverse lookup zones: Create for all internal subnets. Required for PTR records used by logging and some applications.
  • DNS debug logging: Enable temporarily for troubleshooting only — generates significant disk I/O.

9.5 DHCP Standards

  • DHCP failover: Configure hot-standby (primary/secondary) or load-balance mode between two servers. Maximum Client Lead Time (MCLT): 1 hour.
  • Scope design: One scope per VLAN/subnet. Reserve 10-20% of range for static assignments (servers, printers, APs).
  • Lease duration: 8 hours for wired, 4 hours for wireless, 30 minutes for guest networks.
  • DHCP options: Option 6 (DNS servers), Option 15 (DNS suffix), Option 3 (default gateway), Option 44 (WINS — only if still required).
  • Reservations: Use for printers, access points, IoT devices, cameras. Document MAC-to-IP mappings.
  • Authorization: DHCP servers must be authorized in AD. Rogue DHCP detection via monitoring.

9.6 DFS (Distributed File System)

  • DFS Namespaces: Domain-based namespace for simplified UNC paths (\\domain\Share instead of \\Server\Share).
  • DFS Replication (DFSR): Multi-master file replication between servers. Use for shared content, not user home drives (conflict risk).
  • SYSVOL replication: Modern domains use DFSR. Legacy FRS → migrate to DFSR: dfsrmig /SetGlobalState 3.
  • Staging area: Size to hold the largest 32 files replicated. Undersized staging causes excessive re-staging and WAN traffic.
  • Conflict resolution: Last-writer-wins by default. Losing file moved to ConflictAndDeleted folder. Monitor this folder size.
  • Initial sync: Pre-seed with robocopy for large datasets to avoid full replication over WAN.

9.7 Windows Server Update Services (WSUS)

  • WSUS on-prem: Use for environments without Intune/SCCM. Deployment via GPO: Computer → Admin Templates → Windows Components → Windows Update.
  • Upstream/downstream: Central WSUS server syncs from Microsoft Update; branch offices use downstream (replica mode).
  • Classification approval: Critical Updates + Security Updates = auto-approve after 7-day test period. Feature packs, drivers = manual approval only.
  • Maintenance: Run WSUS cleanup wizard monthly. Decline superseded updates. Re-index SUSDB quarterly.
  • WSUS + Intune: Don't use both. If Intune manages updates, set Do not connect to any Windows Update Internet locations GPO to prevent conflicts.

9.8 Hyper-V Standards

  • Hyper-V host: Dedicated role — do not combine with DC, file server, or other roles on production hosts.
  • Generation 2 VMs: Default for all new VMs (UEFI, Secure Boot, VHDX, better performance).
  • Virtual switches: External (VM + host + physical network), Internal (VM + host only), Private (VM-to-VM only).
  • Checkpoints: Use production checkpoints (application-consistent via VSS). Disable standard checkpoints in production.
  • Live migration: Configure Kerberos constrained delegation between hosts. Use SMB for live migration traffic on dedicated VLAN.
  • Replica: Hyper-V Replica for DR — replicates VMs to a secondary site. RPO: 30 seconds, 5 minutes, or 15 minutes.
  • Resource allocation: Never over-commit memory on production hosts. Dynamic memory acceptable for non-critical VMs with proper min/max/buffer settings.
  • Networking: Enable SR-IOV for high-performance network workloads. Use NIC teaming (SET — Switch Embedded Teaming) on 2016+.

9.9 Windows Server Clustering

  • Failover Clustering: Minimum 2 nodes. Requires shared storage (SAN, S2D, or iSCSI). All nodes on same AD domain.
  • Cluster validation: Always run Test-Cluster before creating or modifying. All tests must pass.
  • Quorum: Use disk witness for even-node clusters, cloud witness (Azure Storage) for 2-node clusters, file share witness if no Azure available.
  • Cluster-Aware Updating (CAU): Automates patching with rolling updates. Nodes drain, patch, reboot, rejoin automatically.
  • Storage Spaces Direct (S2D): Software-defined storage using local disks across cluster nodes. Minimum 2 nodes, minimum 4 drives per node.
  • CSV (Cluster Shared Volumes): Single namespace for shared VHDX storage. Required for Hyper-V VMs on clustered storage.

9.10 Active Directory Certificate Services (AD CS / PKI)

  • Two-tier hierarchy: Offline root CA + online issuing CA. Never put the root CA online in production.
  • Root CA: Standalone, offline, issue only subordinate CA certificates. CRL validity: 1 year. Store root cert and CRL in AD.
  • Issuing CA: Enterprise (AD-integrated), online, issues user/computer/server certificates.
  • Certificate templates: Duplicate default templates, never modify originals. Set minimum key size 2048-bit RSA or ECDSA P-256.
  • Auto-enrollment: Configure via GPO for computer certificates (802.1X, IPsec, LDAPS). Requires template permissions + GPO.
  • CRL/OCSP: Publish CRL to AD + HTTP. CDP and AIA must be accessible by all validating entities. OCSP responder for high-volume environments.
  • Key archival: Enable for user encryption certificates (EFS, S/MIME). Do not enable for signing certificates.

9.12 Common MSP Windows Server Failure Patterns

  • SYSVOL not replicating (DFSR journal wrap — Event ID 2213). Fix: non-authoritative restore of DFSR member.
  • AD replication failing after snapshot restore (USN rollback). Fix: Forcefully demote and re-promote the DC. Never restore DC snapshots without proper VM-GenerationID support.
  • Time sync drift causing Kerberos failures. Fix: PDC emulator must sync to external NTP; all others sync to domain hierarchy.
  • LDAP channel binding / signing enforcement breaking legacy apps. Test with diagnostic logging before enforcing.
  • Expired certificates on LDAPS — breaks Entra ID Connect, RADIUS, third-party integrations. Monitor with PKI Health monitoring.
  • Group Policy not applying — check blocked inheritance, denied permissions, loopback processing, WMI filter failures.

11.3 Virtual Networking (VNet)

  • VNet design: Use RFC 1918 address space. Plan for growth — difficult to resize later without disruption.
  • Subnets: Separate subnets for: web tier, app tier, data tier, management, gateway, bastion, private endpoints.
  • VNet peering: Connect VNets within or across regions. Non-transitive — use hub-spoke topology with Azure Firewall or NVA for transit.
  • Hub-spoke topology: Central hub VNet with shared services (firewall, DNS, VPN gateway). Spoke VNets peered to hub.
  • DNS: Azure Private DNS zones for private name resolution. Link zones to VNets. Use for internal service discovery.
  • Service endpoints: Route traffic to Azure PaaS services over the Azure backbone instead of public internet.
  • Private endpoints: Private IP for PaaS services (Storage, SQL, Key Vault) inside your VNet. Preferred over service endpoints.

11.4 Network Security Groups (NSGs)

  • NSGs filter traffic at subnet or NIC level. Stateful — return traffic auto-allowed.
  • Priority: Lower number = higher priority (100-4096). Default rules allow VNet-to-VNet and outbound internet.
  • Best practices:
    • Apply NSGs to subnets, not individual NICs (easier to manage).
    • Use Application Security Groups (ASGs) to group VMs by role instead of IP addresses.
    • Deny all inbound as last rule (already default). Add specific allow rules above.
    • Never open RDP (3389) or SSH (22) to the internet. Use Azure Bastion or JIT VM access.
  • NSG flow logs: Enable for traffic analysis. Send to Log Analytics workspace for queries and visualization.
  • Network Watcher: IP flow verify, next hop, connection troubleshoot, packet capture, NSG diagnostics.

11.5 Azure Firewall

  • Azure Firewall: Cloud-native, managed firewall service. Deploy in hub VNet for centralized traffic control.
  • SKUs: Basic (SMB, low throughput), Standard (threat intelligence, FQDN filtering, network/app rules), Premium (TLS inspection, IDPS, URL filtering, web categories).
  • Rule collections: Network rules (L3/L4), Application rules (L7 FQDN-based), NAT rules (DNAT inbound).
  • Firewall Policy: Use Firewall Policy instead of classic rules. Policies can be inherited (base → child) for multi-tier management.
  • DNS proxy: Enable Azure Firewall as DNS proxy for VNets. Required for FQDN filtering in network rules.
  • Forced tunneling: Route all internet-bound traffic through Azure Firewall. Requires management subnet and public IP.
  • Threat intelligence: Block traffic from/to known malicious IPs and domains. Enable in "Alert and deny" mode.

11.6 VPN & ExpressRoute

  • Site-to-Site VPN: Connect on-prem to Azure VNet. VPN Gateway required in gateway subnet (/27 minimum).
  • Gateway SKUs: VpnGw1 (650 Mbps) for small, VpnGw2 (1 Gbps) for medium, VpnGw3-5 for larger. AZ SKUs for zone-redundant.
  • Point-to-Site VPN: Remote user access to Azure VNet. Supports IKEv2, SSTP, OpenVPN. Certificate or Entra ID authentication.
  • ExpressRoute: Private, dedicated connection to Azure via connectivity provider. 50 Mbps to 100 Gbps. For production workloads requiring guaranteed bandwidth and low latency.
  • Coexistence: ExpressRoute + S2S VPN for failover. VPN as backup when ExpressRoute circuit is down.
  • BGP: Enable BGP on VPN Gateway for dynamic routing with on-prem networks. ASN 65515 reserved for Azure.

11.7 Azure Storage

  • Storage account types: Standard (HDD-backed, cheapest), Premium (SSD, low latency). General Purpose v2 is the default.
  • Services: Blob (object storage), File (SMB/NFS shares), Queue (messaging), Table (NoSQL key-value).
  • Access tiers: Hot (frequent access), Cool (30+ day retention, lower storage cost), Cold (90+ day), Archive (180+ day, offline, hours to rehydrate).
  • Redundancy: LRS (3 copies in one datacenter), ZRS (3 copies across zones), GRS (6 copies across regions), GZRS (ZRS + cross-region).
  • Security:
    • Disable public blob access by default.
    • Use private endpoints for VNet access.
    • Require HTTPS (minimum TLS 1.2).
    • Use Entra ID authentication over shared keys where possible.
    • Enable soft delete for blobs (14-day minimum) and containers.
    • Immutable storage (WORM) for compliance data.
  • Azure Files: SMB 3.0 file shares. Can replace on-prem file servers. Azure File Sync for hybrid caching.

11.8 Azure AD Domain Services (Entra Domain Services)

  • Managed AD DS in Azure. Provides domain join, LDAP, NTLM, Kerberos without deploying DCs.
  • Use cases: Lift-and-shift apps requiring AD DS, LDAP-bound applications, legacy authentication needs.

  • Managed domain: aadds.company.com. Separate from on-prem AD. Join Azure VMs to this domain.

12.7 Defender Incident Response Playbook

SECTION 15 – VSA ARCHITECTURE & AGENT FUNDAMENTALS

15.1 VSA Platform Architecture

  • Kaseya VSA 9.5 is an agent-based Remote Monitoring and Management (RMM) platform. The server (on-prem or SaaS) communicates with lightweight software agents installed on every managed endpoint.
  • Communication: Agent initiates outbound TCP on port 5721 to the VSA server. No inbound firewall rules needed on endpoints.
  • P2P/Remote Control: UDP/TCP 3478 to stun.kaseya.com for NAT traversal and peer-to-peer remote desktop connections.
  • Firewall requirement: Enable Consistent NAT on SonicWall (and similar firewalls) to ensure reliable agent-to-server P2P connectivity.
  • Agent service: Runs as two Windows services — Kaseya Agent (primary communication) and Kaseya Agent Endpoint (endpoint security functions).
  • Agent identity: Each agent is uniquely identified by machineID.groupID.organizationID. This triple determines policy inheritance, scope visibility, and reporting hierarchy.
  • Working directory: C:\kworking (Windows default). Stores temporary operational files — auto-recreates if deleted. Customizable for white-label MSPs.
  • LAN Cache: Designate one machine per site as a file distribution point. Other agents on the same LAN pull files locally instead of from the VSA server — critical for WAN bandwidth savings on large deployments.
  • Each agent consumes one license. Multiple agents CAN run on the same machine (e.g., for testing), each consuming a separate license.

16.2 Deployment Packages — Chain of Life

Chain of Life: Settings → Templates → Packages → Machines. Understand this chain and you understand Kaseya deployment.

16.2.1 Settings

  • Base configuration for agents: check-in interval, working directory, credential profiles, agent menu options.
  • Applied at the group or organization level. All agents in a group inherit these settings.

16.2.2 Templates

  • Templates are "dumb machines" that never check in. They serve as a gold standard for settings.
  • Critical best practice: Always use templates (not real machines) as your settings source. Real machines might get deleted or go offline — templates are permanent and reliable.
  • Create one template per standard configuration: "Standard Workstation", "Server", "Laptop", etc.

16.2.3 Packages

  • The downloadable installer (.exe) that gets deployed to target machines.
  • 7-Step Package Creation:
    1. Assign to group — use the "unassigned" group (sort later)
    2. Silent install config — suppress dialog boxes (Windows only; Mac cannot be silent)
    3. Download and test — verify the package installs correctly before mass deployment
    4. Select settings source — point to a template (not a live machine)
    5. Identify target OS — Windows, macOS, or Linux
    6. Bind admin credentials — only for client-specific packages, never default packages
    7. Name the package — descriptive: "ClientName - Workstation - Windows"

16.2.4 Machines

  • Once the package is installed, the machine checks in and inherits settings from its group/template.
  • New machines land in the "unassigned" group → run initial audit → move to correct group via Change Group.

16.3 LAN Watch (Network Discovery) Deep Dive

  • Probe selection: Use a domain controller or server OS as the probe. Never use Windows XP (limited nmap compatibility).
  • Scan types:
    • Quick Scan: NMAP IP alive detection — fast, finds all responding IPs.
    • DNS Scan: Resolves device names and MAC addresses via reverse DNS and ARP.
    • Deep Scan: Port scanning + OS identification. Takes ~5 minutes per subnet. Most thorough.
  • Scan phases: Installing → Quick Scan → Deep Scan → DNS/MAC lookup (kprtpng reverse DNS + MAC vendor lookup).
  • Discovered Devices Grid: Checkbox = agent-eligible device. Stop symbol = not eligible. Unclassified = use "Change Type" to manually categorize.
  • Enable Harvesting: Discovery → Administration → Settings → Automatic Network Harvesting. Set interval to 1 hour for continuous network awareness.
  • Data Retention: Increase "days to keep unseen devices" from default 7 to 30 days — prevents transient laptops from being re-discovered every time they return to the network.
  • SNMP Discovery: Enable the SNMP tab with community strings (e.g., "public") to discover switches, printers, and other infrastructure.

16.4 Domain Watch (Active Directory Sync)

  • Automatically discovers and syncs Active Directory domains.
  • Auto-deploys agents with correct machine group placement based on AD OU structure.
  • Deploys across subnets and VLANs — unlike LAN Watch, which is limited to the probe's LAN.
  • GPO-based deployment: Creates "KAgentDeployment" Group Policy object. Computers receive agent at next GPO refresh.
  • Tabs: Probe Deployment → Preview Scan → Agent Deployment → Schedule & Status.
  • Full Sync vs. Harvest: Full Sync discovers all AD objects. Harvest runs periodically to catch new machines.
  • Cross-platform rule: Deployment is "like-to-like" — a Windows probe can only deploy Windows agents. Use separate probes for Mac/Linux if needed.
  • Deploy policy: Use "Audit Agent" for initial deployment — safe, non-invasive, performs baseline audit without changing the machine.

16.5 Post-Deployment Onboarding Checklist

  1. Verify agent check-in (green icon, not orange).
  2. Wait for baseline audit to complete (clock overlay disappears).
  3. Move from "unassigned" to correct machine group.
  4. Verify credentials: Agent → Credentials → test connection.
  5. Assign monitor set (or confirm inherited from group).
  6. Assign patch profile (or confirm inherited).
  7. Run initial patch scan.
  8. Run initial agent procedure (standard onboarding script).
  9. Verify in Audit → View Individual Machines for complete hardware/software inventory.
  10. Confirm remote control works (Quick View → test RC session).

17.5.1 Managed Variables (Multi-Client Procedures)

SECTION 18 – MONITORING & ALERTING

18.1 Monitor Sets Architecture

  • Monitor Sets are groups of performance counters, services, processes, and event log entries that VSA checks continuously on each agent.
  • Components of a Monitor Set:
    • Counter Objects: Windows Performance Counter categories (Processor, Memory, PhysicalDisk, Network Interface, etc.)
    • Counter Instances: Specific items within a counter (e.g., "C:" for LogicalDisk, "_Total" for Processor)
    • Thresholds: Warning and alarm values (e.g., CPU > 90% for > 5 min = alarm)
    • Services: Windows service monitoring (running/stopped state)
    • Processes: Process presence monitoring (running/not running)
  • Apply monitor sets to individual machines, groups, or organizations. Inheritance flows downward.

18.2 ATSE Alert Action Codes

ATSE is the core alert engine. Every monitoring trigger can fire any combination of these four actions.
CodeActionDescriptionBest Practice
ACreate AlarmGenerates an alarm entry in the Alarm Log. Appears on Dashboard and in alarm summaries.Enable for all critical and warning thresholds
TCreate TicketAuto-creates a ticket in Service Desk or Ticketing module with alert details.Enable for issues requiring technician action
SRun ScriptExecutes an Agent Procedure automatically when the alert fires. This is auto-remediation.Use for known-fix issues (restart service, clear temp, etc.)
ESend EmailSends notification email to specified recipients.Critical alerts only — avoid alert fatigue

18.2.1 Auto-Remediation Pattern (ATSE-S)

Example: Print Spooler stops → auto-restart

Monitor Set: Windows - Services
  Service: Spooler
  Condition: Service Stopped
  ATSE: A=Yes, T=Yes, S=Yes (→ "Fix – Print Spooler Reset" procedure), E=No

Result: Spooler stops → alarm created → ticket opened → spooler auto-restarted
        If remediation succeeds, ticket can be auto-closed.
        If it fails again within re-arm period, no duplicate alarm floods.

18.3 Auto Learn

  • Automatically adjusts monitoring thresholds based on observed real-world performance over a learning period.
  • Prevents false positives from unrealistic default thresholds (e.g., a server that normally runs at 70% CPU shouldn't alarm at 75%).
  • Learning period: Typically 7-14 days. During this time, Auto Learn observes patterns and sets appropriate baselines.
  • After learning: Thresholds become Individualized Monitor Sets (IND prefix) — per-machine customized thresholds.
  • Best practice: Enable Auto Learn on initial deployment, then review the individualized thresholds and adjust if needed.

18.4 Individualized Monitor Sets

  • Prefixed with IND in the monitor set name. Created by Auto Learn or manually per-machine.
  • Override the group-level monitor set for that specific machine.
  • Use for machines with unique performance profiles (e.g., a database server that legitimately runs at 85% CPU).
  • Review IND sets periodically — they can mask performance degradation if the baseline was set during abnormal conditions.

18.5 Re-arm Time

  • After an alarm fires, the re-arm period is the cooldown before the same alarm can fire again.
  • Minimum recommended: 1 hour. Shorter re-arm periods cause alarm floods that overwhelm technicians and ticket queues.
  • For critical services (DC replication, backup jobs): 15-30 minute re-arm is acceptable.
  • For performance counters (CPU, memory, disk): 1-2 hour re-arm prevents noise.

18.7 Event Log Monitoring

  • Monitor Windows Event Logs for specific Event IDs, Sources, or severity levels.
  • Critical Event IDs to monitor:
    • 41 (System): Unexpected shutdown/bluescreen — hardware or driver issue.
    • 1001 (Application): Windows Error Reporting (WER) — application crash details.
    • 1014 (System): DNS resolution timeout — network or DNS issue.
    • 4625 (Security): Failed logon — potential brute force.
    • 4720 (Security): User account created — detect unauthorized account creation.
    • 4732 (Security): Member added to security-enabled local group — privilege escalation.
    • 7034 (System): Service crashed unexpectedly.
    • 7036 (System): Service entered running/stopped state.
    • 1102 (Security): Audit log cleared — potential tampering.
  • Use ATSE on event log alerts for auto-remediation (e.g., Event 7034 + S = restart crashed service).

18.8 SNMP Monitoring

  • Monitors network infrastructure devices without agent installation: switches, routers, printers, UPS, APs.
  • Setup:
    1. Discovery → Networks → select network → SNMP tab.
    2. Enable SNMP + Enable Standard Monitoring.
    3. Enter community strings (e.g., "public" for read-only).
    4. Discover SNMP devices via LAN Watch.
  • Standard Monitoring: Zero-configuration — auto-monitors common OIDs for uptime, interface status, CPU, memory.
  • Custom SNMP Sets: Add specific OIDs for vendor-specific metrics (SonicWall VPN sessions, UPS battery level, etc.).
  • Topology Map: Interactive network visualization. Right-click SNMP nodes to enable/disable monitoring per device.
  • Device status indicators: Solid green = Online, Solid gray = Offline, Dotted gray = Unknown/never polled.
  • ATSE alerts apply to SNMP monitoring — create tickets for offline switches, email for UPS battery low, etc.

18.9 System Check Monitoring

  • Disk Space: Alert when free space drops below threshold. More responsive than performance counter monitoring.
  • Offline Duration: Alert when agent has been offline for X minutes/hours. Critical for detecting down servers.
  • New Agent Check-in: Alert when a new agent checks in for the first time. Audit trail for rogue deployments.
  • Agent Procedure Failure: Alert when a scheduled procedure fails. Ensures automation reliability.
  • Patch Status: Alert when machines have unapplied critical patches beyond X days.

19.2 Patch Profiles

  • Scan Profile: Defines what to scan for and how often. Schedule daily scans for workstations, weekly for servers (off-hours).
  • Deployment Profile: Controls how patches are deployed — schedule, reboot behavior, pre/post procedures, blackout windows.
  • Alert Profile: Triggers when machines fall out of compliance (e.g., critical patches missing for > 7 days).

19.3 Deployment Best Practices

  • Deployment rings:
    1. Pilot (Day 0-3): IT staff machines, test environment. Immediate deployment.
    2. Early Adopters (Day 3-7): Non-critical workstations. Small group for wider validation.
    3. Broad (Day 7-14): All standard workstations.
    4. Servers (Day 14-21): Production servers during maintenance window.
    5. Critical Infrastructure (Day 21-30): Domain controllers, Exchange, SQL servers.
  • Reboot preferences: Warn user 15 minutes before reboot with countdown. Allow 2 deferrals. Force reboot after 48 hours of deferrals.
  • Blackout windows: Block patching during business hours (8 AM - 6 PM) and month-end processing for accounting firms.
  • Pre/Post procedures: Run backup before patching. Run validation script after patching (check services, verify boot).
  • Force Update: Use sparingly — immediately scans and deploys. For zero-day emergencies only.

19.4 Compliance Monitoring

  • Daily compliance validation: Kaseya checks each machine's scan and deployment status against profile requirements.
  • Scoring: Machines scored 0-100. Critical patches missing > 10 = score 0. Fully patched = score 100.
  • Compliance reports: Schedule weekly compliance reports per organization. Email to client stakeholders and internal management.
  • Dashboard: Patch Management → Dashboard shows patch compliance across all organizations at a glance.

19.5 Third-Party Software Management

  • Supports popular applications: Adobe Reader, Flash (legacy), Java, Chrome, Firefox, 7-Zip, VLC, Notepad++, and more.
  • Auto-update: Configure third-party profile to auto-update managed applications on a schedule.
  • Software deployment: Use Software Management → Software Deployment to push new application installs across machines.
  • Detection: Audit module identifies installed third-party software versions. Cross-reference with known vulnerable versions.

SECTION 22 – SECURITY, ADMINISTRATION & TROUBLESHOOTING

22.1 Users, Roles & Scopes

Setup order matters: Create Role first → Create Scope → Create User → Assign Role + Scope to User.

22.1.1 Roles (Function Access)

  • Roles define what functions a user can access (which modules, which actions within modules).
  • Master Role: Full access to all functions. Assign only to senior administrators.
  • Role types: Kaseya Advanced (full customization), Kaseya Basic (predefined limited access).
  • Customize via Access Rights tab — granular per-module, per-function permissions.
  • Create standard roles: "Helpdesk Tech" (view-only + RC), "Senior Tech" (full agent management), "Admin" (full access except system config), "Auditor" (read-only all data).

22.1.2 Scopes (Data Visibility)

  • Scopes define which data objects (machines, groups, organizations) a user can see.
  • Master Scope: Full visibility of all organizations and machines.
  • Use scopes to restrict MSP technicians to specific clients (Scope "Client A" = only machines in Client A's org).
  • Co-managed services: Create a scope for the client's internal IT team — give them visibility into their own machines only.

22.1.3 User Administration Best Practices

  • Never share logins. Kaseya "does not share well" — two users on the same account overwrite each other's filters and cause forced logouts.
  • Each user needs at least one role + one scope. Multiple roles/scopes can be assigned but only one active at a time.
  • "Inspect What You Expect": After creating a new user, log in as that user to verify the restricted access works correctly before handing over credentials.
  • Individual accounts are essential for audit trail — every action is logged per-user.

22.2 VSA Security Hardening

  • Admin access: Restrict VSA web console to trusted IP ranges. Use Kaseya's built-in IP restriction (System → Configure → Access Control).
  • MFA: Enable multi-factor authentication for all VSA admin accounts. Supports TOTP (Authenticator apps).
  • Session timeout: Configure auto-logout after 15-30 minutes of inactivity.
  • Password policy: Minimum 12 characters, complexity required, 90-day max age, no reuse for 12 generations.
  • API security: Third-party API tokens with configurable start/expiration dates, IP whitelists, and org-level access restrictions.
  • Agent hardening: Disable the agent system tray icon and menu for end users (prevents user from disabling remote control or pausing the agent).

22.3 CVE-2021-30116 — Critical Vulnerability Knowledge

CRITICAL HISTORICAL VULNERABILITY: This was the attack vector for the REvil ransomware supply chain attack in July 2021, affecting ~1,500 organizations worldwide. Understand this to protect your VSA deployment.
  • CVSS Score: 9.8 (Critical)
  • Attack Vector: The /dl.asp endpoint leaked agent credentials without requiring authentication.
  • How it worked:
    1. Agent installation creates C:\Program Files (x86)\Kaseya\[ID]\KaseyaD.ini containing Agent_Guid and AgentPassword.
    2. Credentials were passed via GET request (visible in URLs, logs, browser history).
    3. Attackers submitted extracted credentials to /dl.asp → received a sessionId cookie → gained full authenticated access.
    4. Attackers then pushed malicious agent procedures (ransomware) to all managed endpoints.
  • Remediation (all must be done):
    1. Upgrade to VSA 9.5.7 or later (patches the /dl.asp vulnerability).
    2. Rotate all agent credentials.
    3. Block external access to /dl.asp at the firewall/WAF level.
    4. Enable MFA for all admin accounts.
    5. Implement network segmentation — VSA server should not be directly internet-facing.
    6. Review all agent procedures for unauthorized modifications.
    7. Monitor for IOCs: unusual agent procedure deployments, new admin accounts, unexpected outbound connections.

22.9 Master Troubleshooting Methodology

The Kaseya Troubleshooting Ladder. Follow this order for any issue — it eliminates 90% of problems before you need to escalate.
  1. Check agent status: Is it online (green)? Offline (gray)? Never checked in (orange)? Suspended (red)?
  2. Check filters: Are you seeing all machines? Hit the Reset button — persistent filters are the #1 cause of "missing" machines.
  3. Check connectivity: Can the agent reach the VSA server on TCP 5721? Test: telnet vsaserver.domain.com 5721.
  4. Check credentials: Agent → Credentials. Test the connection. Expired or wrong credentials = procedures fail silently.
  5. Check procedure history: Agent → Agent Logs → Procedure History. Look at pass/fail status and error messages.
  6. Check working directory: Is C:\kworking present and writable? If deleted, it should auto-recreate — if it doesn't, the agent service may need restart.
  7. Check agent services: Are "Kaseya Agent" and "Kaseya Agent Endpoint" services running? Restart if stopped.
  8. Check Windows services: For monitoring issues, verify "Performance Logs & Alerts" service. For patching, verify "Windows Update" service.
  9. Check the logs: Agent Logs → relevant log type. Configuration Changes shows what was modified and when.
  10. Check the firewall: Especially SonicWall — enable Consistent NAT, verify TCP 5721 and UDP/TCP 3478 are allowed.
